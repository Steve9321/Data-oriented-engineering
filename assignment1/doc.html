Overview

This documentation covers a set of REST-based APIs that effectively manage a coffee order processing logic.

Responses

The Task API returns three possible payloads (as JSON object arrays):

  1. Order Payload

  2. Payment Payload

  3. Error Payload

See below for details.

Order Payload

A Order object looks like this:

Single Order Object
{
  "Order" : [
    {
      "ID" : 1521870875,
      "additions" : "sugar",
      "cost" : 3.8,
      "drink" : "Mocha",
      "status" : "waiting",
    }
  ]
}

Order objects MUST be returned as a named array (order). The array MAY have only one member. The valid list of properties for a Order object are: [ID], [additions], [cost], [drink], and [status]. See Data Elements for more information on each property.

Payment Payload

A Payment object looks like this:

Payment Object
{
  "Payment" : [
    {
      "ID" : 1521870875,
      "amount" : 7.6,
      "cardID" : "123456",
      "expiry" : "07/2020",
      "name" : "Tony",
      "number" : 2
    }
  ]
}

Payment objects MUST be returned as a named array (Payment). The array MAY have only one member. The valid list of properties for a Payment object are: [id], [name], [amount], [cardID], [number], and [expiry]. See Data Elements for more information on each property.

Error Payload

When the service encounters an error (HTTP 4xx or 5xx) the service returns an Error Payload that looks like this:

Error Payload
{
  "error" : {
    "ID" : 1521870875,
    "ChangeStatus" : "Not found",
  }
}

CheckPaid

This part is only designed for the barista to check whether this order has already been paid before releasing. Barista only needs to check it which is based on ID from Payment Operations tables.

When constructing a request body for this operation, it SHOULD look like this:

Check Order Paid Request Body
{
  "ID" : 1521870875,
  "PaidResult" : True,
}

Data Elements

What follows is a list of all the possible data elements that MAY appear within a TPS web API payload. Note that these values can appear in both requests and responses.

Note
 This list is in alphabetical order. Note the references in each definition to learn which TPS web API payloads use each data element.

code

HTTP Error code. See Error Payload.

id

The record/object identifier, each order&payment has related id number. it will be created automatically according to the current time. See Order Payload, Payment Payload.

drink

The name of coffee. See Order Payload.

cost

The price of the coffee. See Order Payload.

additions

The extra ingredient of coffee such as sugar, milk, chocolate and shot. See Order Payload.

status

The status of the openorder, only can be changed by barista. The initial value is "open", it changed into " preparing" and "released". See Order Payload.

name

The customer's name. See Payment Payload.

cardID

The card info of customer, the valid card ID should be 16 digits(card only). If the order is paid by cash, this part will be empty. See Payment Payload.

PaymentType

The PaymentType(card or cash) is created by itself in accordance with the input of cardID, See Payment Payload.

number

The number of coffee that the customer order once, See Payment Payload.

amount

The total cost of the bill, it will be calculated by the program automatically, including the additions fee, see payment Payload

expiry

The expiry date of customer's card(card only), if this order is paid by cash, this section should be blank. See Payment Payload.

Extensions

This document describes the Order and Payment service. Any extensions to this service MUST not redefine or change the use/meaning of any URLs, objects (or their properties), arrays, properties, etc. defined in this document. Clients that do not recognize extensions to the service SHOULD ignore them.

Warning
 It is possible that future forward-compatible modifications to this specification will include new elements, attributes, attribute values, and data types. Extension designers should take care to prevent future modifications from breaking or redefining those extensions.

The details of designing and implementing compatible extensions is beyond the scope of this document.
